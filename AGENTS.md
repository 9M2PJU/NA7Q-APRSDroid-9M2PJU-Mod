## 2. What this app is

An Android client for **APRS** (Automatic Packet Reporting System). It does:

- Position reporting (GPS / fixed / periodic / SmartBeaconing).
- Station hub (list of heard stations, sorted by distance).
- Packet log (raw TX/RX).
- APRS messaging (conversations + per-station threads).
- Maps: offline (MapsForge + MBTiles) and online (Google Maps).
- Multiple connection backends: APRS-IS (TCP/UDP/HTTP), AFSK (audio), Bluetooth
  TNC (classic + BLE), USB serial TNC, DigiRig.
- Digipeater and I-gate functionality.
- Profile (JSON) and SSL keyfile (.p12) import via intent filters.

## 3. Build system

- **Language:** Scala 2.11.12 (`org.scala-lang:scala-library:2.11.12`) plus a
  few Java helpers. Built with the
  `gradle-android-scala-plugin` (`jp.leafytree.android-scala`,
  via `com.github.AllBus:gradle-android-scala-plugin:3.5.1`).
- **Android Gradle Plugin:** 3.5.1.
- **SDK:** `compileSdkVersion 33`, `buildToolsVersion 33.0.2`,
  `minSdkVersion 19` (Android 4.4 KitKat), `targetSdkVersion 35`.
- **Java source/target:** 1.8.
- **Build file:** `build.gradle` at the repo root (single-module project --
  there is no `app/` subdirectory). Source sets point at `src/`, `res/`,
  `assets/`, `libs/` directly.
- **Manifest:** `AndroidManifest.xml` at the repo root (not under `app/`).
- **ProGuard:** enabled for both `debug` and `release`
  (`minifyEnabled true`), using `proguard.cfg`.
- **Versioning:** `versionName` = git describe (`grgit.describe()`);
  `versionCode` = `yyyyMMdd` + optional `RELEASE_MINOR` (else `00`).
  Build date and revision are also exposed as string resources
  `build_revision`, `build_date`, `build_version`.
  `APP_VERSION` in `AprsService.scala` is `"APDR" + first 2 digits of
  versionName` -- e.g. versionName "v1.8" → "APDR18". `IgateService.scala`
  displays it as "APRSdroid 1.8" (splits the 2 digits with a dot).
  **Current version tag: `v1.8`** (annotated tag on the release commit).
  To bump the version, create a new `v<x.y>` git tag -- that's all; no
  source changes needed. The CI `release` job triggers on `v*` tags.
- **Google Maps key:** read from `local.properties` as `mapsApiKey`; falls back
  to ge0rg's key (which is restricted to his signing keys and won't work for
  other builds). NA7Q's fork de-emphasizes Google Maps in favor of offline
  maps, but the dependency and manifest entry are still present.
- **Play Store publishing:** `com.github.triplet.gradle:play-publisher:2.4.1`,
  expects `play-publish-credentials.json` and pushes to `beta` track.
- **Release signing:** optional, gated on `RELEASE_KEY_ALIAS` etc. being
  defined (e.g. in `local.properties` or via `-P` flags).

## 8. CI / CD

### GitHub Actions -- `.github/workflows/build.yml`

Triggers: push to `main`/`master`, tags matching `v*`, and manual
`workflow_dispatch`. No PR trigger (signing needs secrets, which fork PRs
can't access). Concurrent runs on the same ref are cancelled in favor of the
newest. The `build` job is also guarded with
`if: ${{ !github.event.pull_request.head.repo.fork }}` for safety.

**`build` job** (ubuntu-latest) -- builds **only a signed release APK** (no
debug APK is produced):
1. Checkout with `submodules: recursive` and `fetch-depth: 0` (full history
   needed -- `versionName` is `grgit.describe()`).
2. JDK 8 (Temurin) -- required by Gradle 5.6.4 / AGP 3.5.1.
3. Android SDK via `android-actions/setup-android@v3`, installs
   `build-tools;33.0.2` and `platforms;android-33`.
4. Writes `local.properties`; sets `mapsApiKey` from the `MAPS_API_KEY`
   secret if present (optional -- falls back to upstream's restricted key).
5. Caches `~/.gradle/caches` and `~/.gradle/wrapper`.
6. Runs `./gradlew test`, then `lintRelease`.
7. Fails early with a clear error if any of the four signing secrets are
   missing.
8. Decodes `RELEASE_KEYSTORE_BASE64` to a temp `.keystore` and runs
   `assembleRelease` with `-P` signing properties
   (`RELEASE_STORE_FILE`, `RELEASE_STORE_PASSWORD`, `RELEASE_KEY_ALIAS`,
   `RELEASE_KEY_PASSWORD`) -- these are the properties `build.gradle` already
   gates the release signing config on.
9. Verifies the APK is actually signed with `apksigner verify --print-certs`.
10. Stages the APK as `artifacts/APRSdroid-9M2PJU-Mod-<tag>.apk` (uses
    `GITHUB_REF_NAME`, falls back to `untagged` on non-tag pushes).
11. Uploads: signed release APK (always), lint reports and test reports
    (always, even on failure -- `if: always()`).

**`release` job** (only on `v*` tags, depends on `build`):
- Downloads the `apk-release` artifact and creates a GitHub Release via
  `softprops/action-gh-release@v2`. Release title is `<tag>-9M2PJU` (e.g.
  `v1.2.3-9M2PJU`), tag stays `<tag>`. `generate_release_notes: true`, signed
  APK (`APRSdroid-9M2PJU-Mod-<tag>.apk`) attached. Has
  `permissions: contents: write`.

### Required GitHub Secrets

All four are **required** -- the workflow fails early if any is missing.

| Secret | Purpose |
| --- | --- |
| `RELEASE_KEYSTORE_BASE64` | base64-encoded `.keystore` / `.jks` file |
| `RELEASE_STORE_PASSWORD` | keystore password |
| `RELEASE_KEY_ALIAS` | key alias inside the keystore |
| `RELEASE_KEY_PASSWORD` | password for that key (same as store password for PKCS12 keystores) |

Optional:

| Secret | Purpose |
| --- | --- |
| `MAPS_API_KEY` | Google Maps Android API key (written to `local.properties` as `mapsApiKey`). If unset, falls back to ge0rg's restricted key -- Google Maps won't work at runtime but the build succeeds. |

### Local keystore (git-ignored)

A release keystore was generated locally under `.dev/secrets/` (git-ignored
via `.dev/`). Contents:

- `release.keystore` -- PKCS12 keystore, RSA 4096, validity 9125 days,
  alias `na7q-aprsdroid-9m2pju`, DN
  `CN=APRSdroid-9M2PJU-Mod, OU=Amateur Radio, O=9M2PJU, L=Kuala Lumpur, ST=Kuala Lumpur, C=MY`.
- `passwords.txt` -- the generated `PASS` (used for both store and key) and
  `ALIAS`.
- `release.keystore.b64` -- base64 of the keystore, ready to paste into the
  `RELEASE_KEYSTORE_BASE64` GitHub Secret.

**Keep `.dev/secrets/` safe.** Back up `release.keystore` somewhere durable
(off-repo, off-machine). If the keystore is lost, you cannot ship updates
that Android will accept as the same app (signing key mismatch).

To regenerate from scratch (only if the existing key is lost/compromised):
```bash
PASS=$(openssl rand -base64 18 | tr -d '/+=' | cut -c1-20)
keytool -genkeypair -v \
  -keystore release.keystore -storepass "$PASS" -keypass "$PASS" \
  -alias na7q-aprsdroid-9m2pju \
  -keyalg RSA -keysize 4096 -validity 9125 \
  -dname "CN=APRSdroid-9M2PJU-Mod, OU=Amateur Radio, O=9M2PJU, L=Kuala Lumpur, ST=Kuala Lumpur, C=MY"
base64 -w0 release.keystore > release.keystore.b64
```

## 9. Key conventions to follow when modifying

- **Scala style:** the existing code uses `_root_.` imports for Android
  classes (e.g. `import _root_.android.app.Activity`). Match that style when
  adding new Scala files in `src/`.
- **Single flat `src/` directory** -- do not introduce package subdirectories
  under `src/` without reason; the project deliberately keeps everything in
  `org.aprsdroid.app` at one level.
- **Intent action strings** are defined as constants in `AprsService.scala`
  (`PACKAGE + ".SOMETHING"`). Reuse these constants instead of hardcoding
  strings; 3rd-party apps depend on them (see `doc/API.mdwn`).
- **Preferences** go through `PrefsWrapper` and are declared in
  `res/xml/preferences*.xml`. New prefs need both the XML entry and a
  `PrefsWrapper` accessor.
- **New activities** must be declared in `AndroidManifest.xml` and should
  follow the inheritance pattern in `activities.md`.
- **Do not** rename the application package -- too many things depend on
  `org.aprsdroid.app` (intent actions, FileProvider authority, profile/keyfile
  intent filters, external API consumers).
- **ProGuard is on for debug too** -- new reflection-based code may need rules
  in `proguard.cfg`.
- **No Google Maps API key in the repo.** Don't commit `local.properties` (it
  is already git-ignored).
- **Git commits: NO co-author trailer.** Do **not** add the
  `Co-Authored-By: Devin <...>` trailer or any `Generated with [Devin]` line
  to commit messages. Keep commit messages clean -- just the subject and body.
  This is a hard rule for every commit in this repo.
- **No em dashes (`--`) on GitHub Pages.** The user does not like double-hyphen
  em dashes in visible text on `docs/index.html`. Use a single hyphen with
  spaces (` - `) instead. This applies to all visible text, meta tags, and
  structured data, but NOT to HTML comments (`<!-- -->`).
- **No GitHub mentions on GitHub Pages.** Do not reference GitHub, GitHub
  Releases, GitHub Actions, or GitHub Pages anywhere in `docs/index.html`
  visible text, meta tags, or structured data. Use "download section on this
  page" or "automated CI/CD pipeline" instead.
- **All notifications use `ToastHelper`.** Always use
  `ToastHelper.show(context, text)` or `ToastHelper.show(context, R.string.xxx)`
  instead of `Toast.makeText(...)`. ToastHelper uses a Snackbar with 1.5s
  duration in Activities (bottom of screen, Material styled) and falls back
  to `Toast.LENGTH_SHORT` (2s) in Service contexts where Snackbar is not
  available.

## 10. Gotchas

- The `jni` symlink at the repo root points to `PacketDroid/jni`. If
  `PacketDroid` is not checked out (submodule not initialized), native builds
  will fail.
- `useLibrary 'org.apache.http.legacy'` is required -- the legacy Apache HTTP
  stack is still used.
- `android.useAndroidX=true` is set in `gradle.properties`, but the
  instrumented test runner still references the old `android.support.test`
  packages. Mixing androidx and support libraries can cause issues; verify if
  touching test setup.
- `targetSdkVersion` is **35** (Android 15). `minSdkVersion` is **19**
  (Android 4.4 KitKat). `compileSdkVersion` remains **33** because the
  Scala build plugin (AllBus `gradle-android-scala-plugin` 3.5.1) is
  incompatible with AGP 8.x+ -- see §13 "Modern Android support &
  limitations" for the full story. `gradle.properties` sets
  `android.suppressUnsupportedTargetSdk=35` to silence AGP 3.5.1's warning.
- `AprsService` declares
  `android:foregroundServiceType="location|microphone|connectedDevice"` and
  the manifest requests `FOREGROUND_SERVICE_LOCATION`,
  `FOREGROUND_SERVICE_MICROPHONE`, and `FOREGROUND_SERVICE_CONNECTED_DEVICE`
  (all required on Android 14+ / API 34+ for per-type foreground services).
  `ServiceNotifier.start()` passes an explicit type bitmask to
  `startForeground()` on API 29+, selecting `microphone` and
  `connectedDevice` only when the corresponding runtime permission
  (`RECORD_AUDIO` / `BLUETOOTH_CONNECT`) is granted. New foreground services
  must follow the same pattern.
- `POST_NOTIFICATIONS` permission is declared (Android 13+). It is added to
  the runtime permission set in `AprsBackend.defaultBackendPermissions()`
  on API 33+ and requested via `PermissionHelper`.
- `BLUETOOTH` and `BLUETOOTH_ADMIN` have `maxSdkVersion="30"` (deprecated on
  API 31+). `BLUETOOTH_SCAN` (with `neverForLocation`) and
  `BLUETOOTH_CONNECT` are declared for API 31+. `READ_EXTERNAL_STORAGE` and
  `WRITE_EXTERNAL_STORAGE` have `maxSdkVersion="32"` (deprecated on API
  33+). `MANAGE_EXTERNAL_STORAGE` is declared for all-files access for
  offline map (MBTiles) files on Android 11+.
- `requestLegacyExternalStorage="true"` is set for Android 10 and below. On
  Android 11+ with `targetSdkVersion 35`, it is ignored; the app must use
  `MANAGE_EXTERNAL_STORAGE` (all-files access) or SAF for offline map file
  access. NA7Q's README documents a "Grant Storage Permissions" flow in the
  OSM Maps preferences.

## 11b. Splash screen

A launch splash screen was added (9M2PJU-Mod branding):

- **Image:** `res/drawable-nodpi/splash.webp` -- full-resolution portrait
  splash (originally `APRSdroid-9M2PJU-Mod-splash.png`, 1536x2752).
  Converted to lossy WebP (quality 85) for size: ~153 KB vs 4.7 MB PNG.
  Lossy WebP is supported since API 14 (our `minSdkVersion`), so no
  compatibility concerns. Placed in `drawable-nodpi` so it isn't
  density-scaled (used as a single asset for all screen sizes).
- **Background drawable:** `res/drawable/splash_background.xml` -- a
  `layer-list` with a black base plus the splash bitmap at
  `gravity="center_crop"` (fills the screen, crops edges on mismatched
  aspect ratios).
- **Theme:** `SplashTheme` in `res/values/themes.xml` -- extends
  `Theme.Holo.NoActionBar`, sets `windowBackground` to
  `@drawable/splash_background`, no title, no action bar.
- **Manifest:** the launcher activity `.APRSdroid` has
  `android:theme="@style/SplashTheme"`. The splash shows as the
  `windowBackground` during cold start while the launcher activity runs
  `onCreate` and routes to Hub/Log/Map via `replaceAct()`. On warm starts
  (activity already in memory) no splash is shown, which is the desired
  behavior.

To replace the splash image, overwrite
`res/drawable-nodpi/splash.webp` (re-export as lossy WebP to keep the APK
small). No code or XML changes needed.

## 11c. App icon & logo

The app icon and logo were replaced with the 9M2PJU-Mod branding
(originally `gemini-clean.png`, 2048x2048 RGBA PNG).

- **Launcher icon (`@drawable/icon`):** referenced by
  `AndroidManifest.xml` (`android:icon="@drawable/icon"`) and
  `ServiceNotifier.scala` (`R.drawable.icon`, used as the small icon for
  message notifications). Density-specific PNGs were generated from the
  2048x2048 source via ImageMagick Lanczos downscaling, matching the
  existing per-density sizes exactly:

  | Density bucket | Path | Size |
  | --- | --- | --- |
  | ldpi | `res/drawable-ldpi/icon.png` | 48x48 |
  | mdpi | `res/drawable/icon.png` | 64x64 |
  | hdpi | `res/drawable-hdpi/icon.png` | 72x72 |
  | xhdpi | `res/drawable-xhdpi/icon.png` | 96x96 |
  | xxhdpi | `res/drawable-xxhdpi/icon.png` | 144x144 |
  | xxxhdpi | `res/drawable-xxxhdpi/icon.png` | 192x192 |

  PNG (not WebP) is used for the launcher icon for maximum launcher/Android
  version compatibility. Files are small (5-54 KB each).

- **Logo (`@drawable/logo`):** `res/drawable-nodpi/logo.png` -- 512x512 PNG,
  for in-app branding (About page, headers) and Play Store listing. Not
  yet referenced by any activity; available as `R.drawable.logo` for
  future use. In `drawable-nodpi` so it isn't density-scaled.

- **Status bar icon (`@drawable/ic_status`):** left unchanged. It's a
  small monochrome silhouette used in the status bar and ongoing-service
  notification; the full-color app icon would look wrong at 24-96px in
  the status bar. `ServiceNotifier.newNotification()` uses
  `R.drawable.ic_status` for the ongoing service notification, and
  `R.drawable.icon` (now the new icon) for message notifications -- both
  correct.

To replace the icon/logo later, regenerate the density-specific PNGs from
a new 2048x2048 source:

```bash
SRC=new-icon.png
magick "$SRC" -filter Lanczos -resize 48x48   res/drawable-ldpi/icon.png
magick "$SRC" -filter Lanczos -resize 64x64   res/drawable/icon.png
magick "$SRC" -filter Lanczos -resize 72x72   res/drawable-hdpi/icon.png
magick "$SRC" -filter Lanczos -resize 96x96   res/drawable-xhdpi/icon.png
magick "$SRC" -filter Lanczos -resize 144x144 res/drawable-xxhdpi/icon.png
magick "$SRC" -filter Lanczos -resize 192x192 res/drawable-xxxhdpi/icon.png
magick "$SRC" -strip -resize 512x512          res/drawable-nodpi/logo.png
```

## 12c. UI modernization (Phase 1)

The app was migrated from `Theme.Holo` (Android 4.0, 2011) to a
**Material 3-inspired design system** built on top of
`Theme.MaterialComponents.DayNight` (Material Components 1.9.0). We can't
use the Material3 library (requires compileSdk 34+; we're at 33), but we
apply M3 design principles: tonal color palette, shape theming, surface
elevation, refined typography, and modern component styles.

### What was done

- **`build.gradle`**: `minSdkVersion` 14 → **19** (required by
  `appcompat:1.6.1` and `material:1.9.0`). Added dependencies:
  - `androidx.appcompat:appcompat:1.6.1`
  - `com.google.android.material:material:1.9.0`
- **`res/values/dimens.xml`** (new): shape corner sizes (4/8/12/16/28dp),
  spacing scale (4/8/16/24/32dp), elevation scale (0/1/3/6dp), action bar
  height (56dp) and elevation (0dp -- flat, modern top app bar).
- **`res/values/colors.xml`** (expanded): full M3-style tonal palette --
  primary/secondary/tertiary with container + on-container variants,
  surface/surfaceVariant, onSurface/onSurfaceVariant, inverseSurface,
  outline/outlineVariant, error with container, system bar colors, scrim.
  Light mode: white surfaces, navy text, amber accent.
- **`res/values-night/colors.xml`** (expanded): matching dark palette with
  elevation tints (darker surfaces at lower elevation, lighter at higher).
  Dark mode: navy surfaces `#0D182D`/`#13203A`/`#1C2F51`, amber accent
  `#CEB619`, cool grey text. Matches the app icon.
- **`res/values/bools.xml`** + **`res/values-night/bools.xml`** (new):

### Why not compileSdk 36 (Android 16)

Compiling against API 36 requires AGP 8.9.1+, which requires Gradle 9.x
and JDK 17. The project's Scala compilation depends on
`com.github.AllBus:gradle-android-scala-plugin:3.5.1`, which only supports
AGP 3.5.x. **No maintained gradle-android-scala-plugin supports AGP 8.x/9.x**
as of 2026 (checked saturday06, xelnaga, xingda920813, AllBus forks -- all
top out at AGP 2.3.3-3.5.x). The Scala-on-Android tooling ecosystem is
effectively unmaintained.

To reach compileSdk 36, one of these would be needed (all are large
efforts outside the scope of this change):
1. Port `gradle-android-scala-plugin` to AGP 8.x/9.x (deep Gradle internals).
2. Rewrite the ~6,800 lines of Scala as Kotlin or Java.
3. Compile Scala with standalone `scalac` and feed class files into AGP
   manually (fragile, no incremental compilation).

### Known limitations on Android 14+ / 15+

These are runtime concerns not fully addressed by this change -- document
and test on real devices:

- **Edge-to-edge enforcement (Android 15+, API 35+):** With targetSdk 35,
  the system forces edge-to-edge display. The app uses `Theme.Holo` (pre-
  Material) throughout. Status bar / navigation bar may overlap content in
  some activities. Proper fix: handle `WindowInsets` in each activity or
  set `fitsSystemWindows="true"` on root views. The splash screen is
  unaffected (uses `center_crop` which fills the full window). This is a
  per-activity code change across `HubActivity`, `LogActivity`,
  `StationActivity`, `MessageActivity`, `ConversationsActivity`, `MapAct`,
  `GoogleMapAct`, and the `*Prefs` activities.

- **16 KB page size (Android 15+, API 35+):** Android 15 devices with 16 KB
  page size require native libraries (`.so` files) to be aligned to 16 KB
  boundaries. The project bundles native libs via `libs/` (jniLibs) and the
  `PacketDroid/jni` submodule. If those `.so` files were built with 4 KB
  page alignment only, the app will fail to load on 16 KB-page devices
  (e.g. Pixel 8+ with Android 15). Fix: rebuild native libs with
  `-Wl,-z,max-page-size=16384` linker flag. Verify with
  `zipalign -c -v 4 <apk>` and check for 16384-byte alignment of `.so`
  entries. This is a native build change, not a Java/Scala change.

- **Foreground service launch restrictions (Android 14+, API 34+):** Apps
  targeting API 34+ cannot start a foreground service from the background
  (with limited exceptions). `AprsService` is started from the UI (boot
  receiver uses `RECEIVE_BOOT_COMPLETED` which is an exempt foreground
  context), so this should be fine -- but verify the boot-start path works
  on Android 14+ devices.

- **`MANAGE_EXTERNAL_STORAGE` is a special permission:** It cannot be
  requested via `requestPermissions()` -- the user must grant it via system
  settings (`Settings.ACTION_MANAGE_APP_ALL_FILES_ACCESS_PERMISSION`). The
  app's "Grant Storage Permissions" preference should launch this intent on
  API 30+. This is a code change in the storage-permission preference
  handler that has not yet been made.

- **`BluetoothAdapter.getDefaultAdapter()` is deprecated on API 31+:** It
  still works but the recommended replacement is
  `BluetoothManager.getAdapter()`. Used in `BluetoothTnc.scala` and
  `BluetoothLETnc.scala`. Not breaking, but may stop working in future
  Android versions.

## 14. Window inset handling -- v2.0.4 work (Android 15/16)

### Problem

On Android 15+ (targetSdk 35), the system forces **edge-to-edge** display.
Content draws under the status bar, navigation bar, and display cutout.
The app had multiple issues:

1. **Double top padding** on PreferenceActivity screens -- the system
   applied the status bar inset via the internal LinearLayout's
   `fitsSystemWindows="true"`, AND the manual `OnApplyWindowInsetsListener`
   applied it again.
2. **Content bleeding under the status bar** -- some activities didn't
   handle insets at all.
3. **PreferenceScreen dialog sub-screens** (Notifications, Connection
   Preferences, etc.) -- shown as `AlertDialog` dialogs with their own
   window. Section headers like "Incoming Messages" overlapped with the
   status bar clock.
4. **Clear Log UI refresh bug** -- after clearing the log, the ListView
   didn't refresh until the user switched tabs and back.

### Solution

#### PreferenceActivity inset handling (`UIHelper.scala`)

- `applyPrefActivityInsets(act)`: enables edge-to-edge
  (`setDecorFitsSystemWindows(false)`), installs an
  `OnApplyWindowInsetsListener` on `android.R.id.content` that reads
  top/bottom insets, applies them as padding to the ListView
  (`android.R.id.list`), and **consumes** the insets
  (`insets.consumeSystemWindowInsets()`) so the internal LinearLayout's
  `fitsSystemWindows` doesn't double-apply them.
- `applyPrefListPadding(act)`: resource-based fallback that reads
  `status_bar_height` and `navigation_bar_height` from system resources.
  Used on OEM ROMs where the inset listener may not fire reliably.
- **`clipToPadding`**: uses the default `true`. Setting `false` allowed
  items to scroll through the padding area under the status bar, causing
  section headers to overlap with the status bar clock.

#### PrefsAct.scala

- `applyPrefTopInset()`: resource-based fallback that applies top+bottom
  padding to the ListView. Re-applied after sub-screen navigation via
  `onContentChanged`, `onWindowFocusChanged`, and `OnPreDrawListener`.
- `onPreferenceTreeClick`: posts delayed `applyPrefTopInset` calls (100ms,
  300ms) to catch sub-screen content swaps.

#### PreferenceScreen dialog sub-screens (`res/values/styles.xml`)

- Added `android:fitsSystemWindows=true` and
  `android:windowDrawsSystemBarBackgrounds=false` to
  `ThemeOverlay.AppTheme.Dialog`. This makes the dialog's window apply
  system window insets as padding, pushing content below the status bar.
- **Android 16 blocks all reflection on hidden APIs** --
  `PreferenceScreen.mDialog`, `Activity.getWindows()`, and
  `WindowManagerGlobal.getWindowViews()` are all blocked
  (`NoSuchFieldException` / `NoSuchMethodException`). Runtime approaches
  to access the dialog's ListView don't work. The theme-based approach
  is the only viable solution.

#### Clear Log refresh (`LogActivity.scala`)

- Override `onOptionsItemSelected` for `R.id.clear`: after
  `StorageCleaner` completes, directly call `locReceiver.startTask(null)`
  to force a cursor reload. The broadcast in `StorageCleaner.onPostExecute`
  may be absorbed by `LocationReceiver2`'s `pending` counter, causing the
  UI to not refresh until the user switches tabs.

### Files modified

- `src/UIHelper.scala` -- `applyPrefActivityInsets`, `applyPrefListPadding`
- `src/PrefsAct.scala` -- `applyPrefTopInset`, `onPreferenceTreeClick`,
  `onContentChanged`, `onWindowFocusChanged`
- `src/LogActivity.scala` -- `onOptionsItemSelected` for Clear Log
- `src/PrefSymbolAct.scala` -- edge-to-edge + inset handling
- `res/layout/prefsymbol.xml` -- removed `fitsSystemWindows="true"`

### Winlink (WLNK-1) -- Stateful APRSLink email gateway

- **Service:** `src/WinlinkService.scala` (singleton tied to AprsService)
- **Protocol:** [APRSLink](https://winlink.org/APRSLink) -- send/receive
  Winlink email via APRS messages to WLNK-1
- **Login flow:** Send "L" → WLNK-1 responds with "Login [XXX]:" challenge
  (3 digit positions) → compute response (3 password chars + 3 random,
  shuffled) → WLNK-1 responds "Hello CALLSIGN" → logged in
- **Session states:** LOGGED_OUT, LOGIN_STARTED, CHALLENGE, LOGGED_IN, ERROR
- **Auto-login:** `AprsService.onPosterStarted()` checks

  against re-triggering when already logged in or login in progress.
- **Login timeout:** 2 minutes -- if stuck in LOGIN_STARTED or CHALLENGE,
  `getState()` auto-resets to LOGGED_OUT so buttons work again.
- **Session timeout:** 2 hours (per Winlink spec)
- **Email composition:** Multi-line body split into ≤67-char APRS messages,
  sent with 500ms delay between lines, terminated by `/EX`
- **Commands:** L (list), R# (read), SP (send), K# (kill), Y# (reply),
  B (logout), H (help), SMS, A (alias), AL (list aliases), F# (forward),
  G# (gateway info), I (info), P (playback)
- **Setup:** Preferences → Messaging → set callsign + Winlink password,
  enable Auto-login
- **DB:** `StorageDatabase.addWinlinkMessage()` stores messages with
  `TYPE_WINLINK_IN` / `TYPE_WINLINK_OUT`

5. Download with: `gh run download <run-id> --name apk-release --dir /tmp/apk-release`
6. Install to phones: `adb -s <device> install -r /tmp/apk-release/APRSdroid-9M2PJU-Mod-master.apk`

### Lessons learned in this session

1. **Android 15+ edge-to-edge:** `navigationBarColor` is ignored. You
   MUST call `setDecorFitsSystemWindows(false)` and let your window
   background extend behind system bars. NOT calling it leaves a gap
   that the system fills with its default light color.

2. **Full-screen splash:** Use `WindowInsetsController.hide(
   navigationBars)` with `BEHAVIOR_SHOW_TRANSIENT_BARS_BY_SWIPE` on
   API 30+ to hide the nav bar during splash. Restore in `replaceAct()`
   before navigating to the target activity.

3. **Winlink auto-login loop:** `onPosterStarted()` fires on every
   position beacon, not just startup. Always check `isLoggedIn` and
   current state before calling `login()`. Add a login timeout (2 min)
   so stuck states auto-recover.

4. **GridLayout button visibility:** Buttons with `layout_height="0dp"`
   + `layout_rowWeight="1"` inside a `wrap_content`-height GridLayout
   get zero height (nothing to divide). Use `wrap_content` height with
   `fill_horizontal` gravity and `minHeight` instead.

5. **XML comments and em dashes:** `--` is invalid inside XML comments.
   When replacing em dashes with `--`, also fix XML comments to use
   single `-` to avoid build failures.

6. **Tag deletion:** If you push the wrong version tag, delete it
   locally (`git tag -d <tag>`), delete remote (`git push origin
   :refs/tags/<tag>`), cancel the CI run (`gh run cancel <id>`), then
   create the correct tag.
